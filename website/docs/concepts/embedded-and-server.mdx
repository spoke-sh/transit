# Embedded And Server Modes

Transit has one engine and two deployment shapes.

## Compare The Modes

| Question | Embedded mode | Server mode |
|----------|---------------|-------------|
| who calls the engine? | your application or runtime directly | remote clients through the daemon |
| where does authority live? | the in-process engine instance | the server boundary and its configured storage |
| how do you inspect it? | local proofs and `transit status` | remote operator commands plus local proof roots on the server node |
| what must stay the same? | lineage, durability labels, manifests, restore behavior | the exact same semantics |

## Embedded Mode

Embedded mode means your process calls the engine directly.

This is the shortest path to understanding:

- append semantics
- replay behavior
- branch and merge lineage
- publication and restore

## Working Plane And Published Plane

Transit now makes the storage split explicit instead of leaving it implicit in
one mutable manifest file.

Working plane:

- local mutable files such as `active.segment` and `state.json`
- current append progress and restart bookkeeping
- not the published discovery surface for acknowledged shared history

Published plane:

- immutable segment objects
- immutable manifest snapshots
- one mutable `frontiers/latest.json` pointer that tells readers which manifest snapshot is current

On a local filesystem-backed node, that published plane lives under
`published/streams/<id>/segments`, `manifests`, and `frontiers/latest.json`.
On a remote object store, the same namespace shape lives under the configured
prefix. The filesystem backend and remote backends now share the same
published-storage concepts.

## Server Mode

Server mode means a daemon exposes those same semantics over the network.

The key point is not just transport. It is preservation.

The server should preserve:

- the same append ordering rules
- the same single-stream batch append semantics, with one batch acknowledgement
  and ordinary per-record replay
- the same opt-in retention semantics, with no implicit default policy and
  whole-segment aging instead of compaction
- the same immutable segment-compression behavior, with `zstd`-compressed
  rolled segments decoded back into normal records on replay
- the same checkpoint-and-resume semantics for derived state, including hosted
  cursors and hosted materialization checkpoints for external-daemon consumers
- the same meaning of append, replay, and tail even when hosted connection I/O
  timeouts are raised for operational robustness
- the same shared-engine semantics while accepted server connections are handled
  concurrently
- the same lineage semantics
- the same storage and durability boundaries
- the same working-plane versus published-plane split, with only
  `frontiers/latest.json` acting as the mutable published discovery boundary

That includes the full failover stack in the shared engine: controlled handoff, automatic leader election, quorum durability, and cluster membership. The one-engine guarantee means these capabilities work identically whether the engine is embedded or behind a server daemon. Multi-primary behavior remains explicitly out of scope.

The common hosted operator path is:

- `transit streams`
- `transit produce`
- `transit consume`

The lower-level `transit server ...` namespace remains available when you need protocol-shaped operations such as lineage inspection, explicit branching, or tail-session control.

That lower-level namespace also carries the hosted batch append shape for one
stream through repeated `--payload-text` values on `transit server append`.
It also carries runtime timeout tuning for the daemon and hosted proofs without
creating a second semantic world for server deployments.
The transport layer can become more tolerant and more concurrent; the record,
lineage, and durability model is still the same engine.
Compression follows the same rule. It is a shared-engine storage choice at the
sealed-segment boundary, not a second hosted message format.
Hosted materialization follows the same rule too: the server exposes cursor,
checkpoint, and resume primitives over the boundary, but authoritative replay
still lives in the shared engine and materialized state does not become part of
stream truth.

That one-engine rule also applies to retention:

- retention is explicit per stream and defaults to `none`
- only whole rolled segments are eligible to age out
- the active segment is preserved
- `retained_start_offset` marks the earliest replayable offset after trimming

Once that retained frontier advances, replay is bounded. That is the intended
operator signal that old history has aged out of the replay surface. It is not
Kafka-style compaction, and it does not mean Transit is selectively erasing
individual subjects inside retained history.

## What Transit Tries To Avoid

Transit is intentionally trying to avoid a split-brain architecture where:

- the library is one product
- the server is another product
- and the semantics drift over time

That is why so much of the repo language insists on **one engine**.

## Practical Reading Rule

If a server guide sounds like it describes different storage semantics than the embedded guide, treat that as a bug in the docs or the implementation.
