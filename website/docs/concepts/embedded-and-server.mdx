# Embedded And Server Modes

Transit has one engine and two deployment shapes.

## Compare The Modes

| Question | Embedded mode | Server mode |
|----------|---------------|-------------|
| who calls the engine? | your application or runtime directly | remote clients through the daemon |
| where does authority live? | the in-process engine instance | the server boundary and its configured storage |
| how do you inspect it? | local proofs and `transit status` | remote operator commands plus local proof roots on the server node |
| what must stay the same? | lineage, durability labels, manifests, restore behavior | the exact same semantics |

## Embedded Mode

Embedded mode means your process calls the engine directly.

This is the shortest path to understanding:

- append semantics
- replay behavior
- branch and merge lineage
- publication and restore

## Server Mode

Server mode means a daemon exposes those same semantics over the network.

The key point is not just transport. It is preservation.

The server should preserve:

- the same append ordering rules
- the same single-stream batch append semantics, with one batch acknowledgement
  and ordinary per-record replay
- the same lineage semantics
- the same storage and durability boundaries

That includes the full failover stack in the shared engine: controlled handoff, automatic leader election, quorum durability, and cluster membership. The one-engine guarantee means these capabilities work identically whether the engine is embedded or behind a server daemon. Multi-primary behavior remains explicitly out of scope.

The common hosted operator path is:

- `transit streams`
- `transit produce`
- `transit consume`

The lower-level `transit server ...` namespace remains available when you need protocol-shaped operations such as lineage inspection, explicit branching, or tail-session control.

That lower-level namespace also carries the hosted batch append shape for one
stream through repeated `--payload-text` values on `transit server append`.

## What Transit Tries To Avoid

Transit is intentionally trying to avoid a split-brain architecture where:

- the library is one product
- the server is another product
- and the semantics drift over time

That is why so much of the repo language insists on **one engine**.

## Practical Reading Rule

If a server guide sounds like it describes different storage semantics than the embedded guide, treat that as a bug in the docs or the implementation.
