# Board Model

Transit is a **lineage-aware append-only log** built on a shared storage engine that runs in two delivery modes:

- **embedded** in-process (library/API surface)
- **server** (network boundary around the same engine)

The first mental model is that a stream is an append-only timeline with explicit ancestry. Branching and merging are storage operations, not application conventions.

## Core primitives

| Primitive | What it means |
|---|---|
| `record` | Immutable payload plus metadata and assigned stream position |
| `stream` | Ordered append history for one identity |
| `branch` | A real child stream that shares parent history to a fork point |
| `merge` | Explicit reconciliation of two or more heads |
| `lineage` | The directed graph of branches and merges |
| `segment` | Immutable chunk of records |
| `manifest` | Metadata that maps lineage state to local/remote segments |
| `checkpoint` | Verifiable anchor for a stream head or derived view |

## Storage shape

Transit keeps durability explicit:

- `local`: persists to local durable state
- `replicated`: makes handoff durable for read-only catch-up and promotion
- `quorum`: requires majority acknowledgement
- `tiered`: persists canonical segment history to the remote tier

The practical split is:

- hot path writes and tailing stay close to local state
- older immutable segments and lineage metadata publish into remote storage as part of the normal model
