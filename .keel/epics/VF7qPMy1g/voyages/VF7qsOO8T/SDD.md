# Launch Public Docs Platform - Software Design Description

> Ship the first public Transit docs surface with a strong product narrative, runnable onramp tracks for library and server users, foundational contract publishing, and a static S3/CloudFront deploy path.

**SRS:** [SRS.md](SRS.md)

## Overview

This voyage introduces a `website/` workspace powered by Docusaurus and treats documentation as a real product surface instead of an afterthought to the repo root. The launch slice has three coupled outputs:

1. A public site scaffold with a Transit-specific look, information architecture, and landing page.
2. Curated MDX docs for concepts plus separate library/server first-run tracks.
3. A reference-publishing and deployment path that carries foundational repo documents into the site and ships static output to S3/CloudFront.

## Context & Boundaries

Transit already has strong Markdown contracts in the repo root, but they are written for maintainers and contributors working directly in the codebase. This voyage adds a second layer: a public documentation experience that introduces the product gradually, keeps the transport metaphor front and center, and still links back to the canonical contracts.

```
┌────────────────────────────────────────────────────────────┐
│                  Transit Docs Voyage                       │
│                                                            │
│  Docusaurus site  Narrative docs  Reference sync  Deploy   │
│                                                            │
│   website/src      website/docs    website/scripts .github │
└──────────────────────────────┬─────────────────────────────┘
                               │
        ┌──────────────────────┼──────────────────────┐
        │                      │                      │
   Repo root docs         npm/node build          S3 + CloudFront
```

Out of scope: autogenerated API reference, docs search, localization, or bespoke AWS infrastructure provisioning.

## Dependencies

| Dependency | Type | Purpose | Version/API |
|------------|------|---------|-------------|
| Docusaurus | npm dependency | Static docs framework, MDX pipeline, theme runtime | 3.x |
| Node.js | runtime | Build and local dev server for the docs workspace | >=20 |
| Root Markdown contracts | repo files | Canonical foundational documents to publish alongside narrative docs | current repo |
| GitHub Actions | CI runtime | Build and deployment automation | current repo |
| AWS S3 + CloudFront | hosting target | Static site delivery and cache invalidation | configured by repo variables/secrets |

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Site location | Create a standalone `website/` workspace | Keeps frontend dependencies isolated from the Rust workspace while matching upstream Keel structure. |
| Canonical/reference model | Keep root docs as the source of truth and sync selected files into generated website docs | Avoids split-brain documentation while still publishing the contracts through Docusaurus. |
| Narrative structure | Lead with product story, concepts, and two adoption tracks (`library` and `server`) | Matches the actual first-time user problem instead of exposing internal maintainer order. |
| Visual direction | Use rail/transport metaphors, signal colors, and printed-timetable style structure | Makes the Transit identity legible and distinct from upstream Keel while fitting the product name. |
| Deployment shape | Build static assets in CI and deploy with `aws s3 sync` plus CloudFront invalidation | Simple, static, and aligned with the requested hosting model. |

## Architecture

The docs launch is a small static-publishing pipeline:

- `website/`
  Docusaurus application with docs content, homepage, theme assets, and local scripts.
- `website/scripts/sync-foundational-docs.mjs`
  Copies selected root Markdown contracts into generated docs pages before build.
- `.github/workflows/docs.yml`
  Installs Node dependencies, runs the sync/build steps, and deploys static output when configured.
- `Justfile`
  May expose a simple local helper so maintainers can build docs without memorizing npm commands.

## Components

### Site Shell

- `website/docusaurus.config.ts`
- `website/sidebars.ts`
- `website/src/pages/index.tsx`
- `website/src/css/custom.css`

Purpose:
Define the site identity, navigation, footer, metadata, and home page.

### Narrative Docs

- `website/docs/intro.mdx`
- `website/docs/concepts/*.mdx`
- `website/docs/start-here/*.mdx`

Purpose:
Teach the product in a user-facing order: what Transit is, how the model works, and how to get started as a library or server user.

### Reference Sync

- `website/scripts/sync-foundational-docs.mjs`
- generated docs under `website/docs/reference/`

Purpose:
Publish foundational repo documents without duplicating them by hand.

### Deployment Pipeline

- `.github/workflows/docs.yml`

Purpose:
Build, verify, and deploy the static site to S3/CloudFront using environment-provided configuration.

## Interfaces

Primary operator interfaces:

- `npm --prefix website run build`
- `npm --prefix website run start`
- `npm --prefix website run sync:foundations`
- GitHub Actions `docs` workflow inputs via repository variables/secrets

Expected deploy configuration:

- site URL / base URL environment variables for Docusaurus
- S3 bucket name
- CloudFront distribution ID
- AWS role or credentials exposed to the workflow

## Data Flow

1. The maintainer edits public MDX content and root foundational documents in the repo.
2. The sync script copies selected root contracts into generated `website/docs/reference/` pages.
3. Docusaurus builds the combined narrative + reference site into static HTML/CSS/JS.
4. CI uploads the built site to S3 and invalidates CloudFront so the new docs become visible.
5. Readers navigate the public docs from introductory story to first-run guides to canonical reference material.

## Error Handling

| Error Condition | Detection | Response | Recovery |
|-----------------|-----------|----------|----------|
| Root docs drift out of publish set | Missing reference pages or broken sidebar links during build review | Fail the sync/build step | Update the sync manifest and rerun build |
| Public docs imply incorrect semantics | Manual review catches contradictions with foundational contracts | Block story acceptance | Rewrite the docs to match canonical terms and guarantees |
| Docusaurus build breaks on missing content or links | `npm run build` fails | Stop release path | Fix content, routing, or config before commit |
| Deploy configuration is absent | Workflow step lacks bucket/distribution settings | Build succeeds, deploy step remains gated or fails clearly | Add repo variables/secrets and rerun |
